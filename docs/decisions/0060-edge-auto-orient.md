# 0060. 엣지 자동 방향 정렬 — 반대로 그어도 유효한 방향으로

- Status: Accepted
- Date: 2026-07-10
- Deciders: 차니, Claude

## Context

차니 보고: "ALB → EC2 연결이 허용되지 않는다고 나와." 규칙(`canConnect('alb','ec2')`)은
**참**이고 올바른 방향의 드래그는 정상 작동한다(브라우저 재현 확인). 문제는 **핸들
방향**이다: ALB는 소스 핸들(오른쪽, EC2로 포워딩)과 **타깃 핸들(왼쪽, CloudFront/
ACM/WAF가 앞단으로 붙음, ADR 0056)**을 **둘 다** 가진다. EC2가 ALB 왼쪽에 있으면
사용자는 자연히 ALB의 왼쪽(=타깃) 핸들에서 EC2 쪽으로 끌게 되고, React Flow는 이를
`EC2 → ALB`로 해석해 `canConnect('ec2','alb')`=거짓 → 거부한다. 의도("ALB가 EC2로
간다")와 그려진 방향이 어긋나는 것이다.

## Decision

**`onConnect`에서 방향을 자동 정렬한다:** 그려진 `source → target`이 불법이지만
`target → source`가 합법이면 조용히 뒤집어 유효한 엣지를 만든다(핸들도 함께 스왑).

- 이 게임의 리소스 계층은 사실상 단방향 DAG라, 한 쌍에서 두 방향이 모두 합법인
  경우가 없다(예: `alb→ec2`만 참, `ec2→alb`은 거짓). 따라서 자동 스왑이 의도를
  왜곡할 위험이 없다. RDS→RDS 복제는 동일 타입이라 스왑 대상이 아니다.
- 양방향 모두 불법일 때만 기존 거부 토스트를 띄운다.

기각한 대안: (a) ALB의 타깃 핸들 숨기기 — CloudFront/ACM/WAF 앞단(0056)이 깨진다;
(b) 안내 문구만 개선 — 근본 마찰(가까운 핸들을 잡는 본능)을 못 없앤다.

## Consequences

- **좋은 점**: "ALB를 EC2로 잇는다"가 어느 핸들에서 끌든 성립한다. 다중 핸들
  리소스(ALB·API GW 등)의 방향 혼동이 사라진다.
- **나쁜 점 / 한계**: 진짜로 반대 방향을 의도한 드문 경우에도 조용히 뒤집힌다 —
  하지만 반대 방향은 어차피 불법이라 실질적 손해는 없다. 로직은 `onConnect`에
  있어 순수 함수 유닛테스트 대상이 아니며 브라우저 재현으로 검증했다.
