# 0034. i18n 유보 재검토 — Korean-first 유지

- Status: Accepted (revisits [ADR 0008](0008-korean-first-ui-no-i18n.md))
- Date: 2026-07-09
- Deciders: 차니, Claude

## Context

[ADR 0008](0008-korean-first-ui-no-i18n.md)은 MVP 속도를 위해 UI 문자열을 한국어로 하드코딩하고
i18n 프레임워크를 도입하지 않기로 했다. Sprint D에서 "영문 재도입 vs 유지"를 정식으로 재검토한다.

재검토 시점의 근거:

- **수요 근거 부족**: 정적 GitHub Pages라 방문자 언어권을 계측할 파이프라인이 없다. 영어권
  사용자가 유의미하게 유입된다는 증거가 아직 없다.
- **도입 비용**: 지금 UI 문자열은 컴포넌트 곳곳에 인라인 한국어다. i18n을 넣으려면 키 추출·
  카탈로그·언어 감지·번역 유지보수라는 지속 오버헤드가 생긴다 — 1인+에이전트 개발 속도에 직접 타격.
- **국제 접근성 부분 확보**: README·README.ko가 이미 **이중언어**라 오픈소스 방문자는 개요·실행
  방법을 영어로 이해할 수 있다. 즉 "발견 가능성"은 코드 i18n 없이도 어느 정도 확보돼 있다.

## Decision

**지금은 Korean-first를 유지한다(코드 변경 없음). 다만 도입 조건을 명시적으로 기록한다.**

다음 **국제화 트리거** 중 하나가 관측되면 i18n 도입을 재검토(=이 ADR을 superseding하는 새 ADR):

1. GitHub 스타가 **100개**를 넘고, 그중 영어권 사용자 관심이 이슈/논의로 드러날 때
2. **영어권 이슈·PR·번역 기여 제안**이 유입될 때
3. 학습 콘텐츠·데모로 **비한국어권 대상 배포**가 제품 목표가 될 때

도입 시 우선 접근(로드맵): (a) 문자열을 컴포넌트에서 카탈로그로 추출 → (b) `lang` 감지 + 수동
토글 → (c) 경량 런타임(예: 자체 `t()` + JSON 카탈로그, 프레임워크 없이) 부터. React 생태계
i18n 라이브러리는 번들·복잡도 대비 효용을 그 시점에 재평가.

## Consequences

- **좋은 점**: 개발 속도를 지키면서 "언제·왜 국제화할지"를 문서로 확정 — 다음에 다시 논쟁하지
  않는다. README 이중언어로 최소 국제 접근성은 유지.
- **나쁜 점 / 한계**: 그 전까지 비한국어권 사용자는 UI를 온전히 쓰기 어렵다. 트리거는 정량
  계측이 아니라 정성 신호(이슈 유입 등)에 기대므로, 판단 시점에 사람이 개입해야 한다.
- **후속 영향**: 트리거 충족 시 문자열 추출이 첫 작업이 된다 — 그때를 대비해 신규 UI 문자열은
  가능하면 컴포넌트 상단 상수로 모아 두면 추출 비용이 준다(강제는 아님).
